# GitHubとは（ざっくり一言で）

## この章でできるようになること

- GitHub が「何のための場所か」を説明できる
- Git と GitHub の役割の違いを言葉にできる
- 会話で出る GitHub 機能（PR / Issue / Review）の意味がわかる

## GitHubとは

GitHub は **Gitの履歴（リポジトリ）を置いて共有するサービス**です。

イメージは「履歴付きの共有フォルダ」ですが、ポイントは **共有 + 合意形成**に強いことです。

- **共有**: チームで同じ履歴を参照できる（バックアップにもなる）
- **合意形成**: 変更をレビューしてから取り込める（Pull Request）
- **進捗管理**: タスクをIssueとして整理できる

## GitとGitHubの違い

- **Git**: 変更履歴を記録する仕組み（道具）
- **GitHub**: その履歴を共有・レビュー・運用する場所（サービス）

## よく出るGitHub用語（最低限）

- **リモート（remote）**: GitHubなどネット上のリポジトリ
- **origin**: そのリモートにつけることが多い標準的な名前
- **Pull Request（PR）**: 変更を取り込んでほしいという提案 + レビューの場
- **Review**: PRに対する確認（OK/指摘/質問）
- **Issue**: タスクやバグ、相談を記録するチケット

## Pull Request は「お願い＋説明＋相談」

PR は単に「差分を見せる」だけでなく、

- 何を変えたか（目的）
- どこを見ればいいか（ポイント）
- どう確認すればいいか（確認観点）

を共有するためのコミュニケーションの器です。

## 非エンジニアが知っておくと会話が楽になること

- 「まだ作業中」なら **draft PR** を使うことがある
- 「この変更はどのIssueに紐づく？」という会話がある
- 「誰がマージする？いつリリース？」の線引きがある（運用ルール次第）
