# レビュー用語集（会話のため）

目的: Pull Request（PR）レビューの会話でよく出る言葉を、非エンジニアでも誤解なく理解できるようにする。

## コメントの温度感

- **LGTM**
  - 「Looks Good To Me」= ひとまずOK（ただし運用によっては“最終承認”ではないことも）
- **nit / nits**
  - 小さな指摘（好み/スタイル/軽微な改善）。致命的ではないニュアンス
- **FYI**
  - 参考までに（必須対応ではないことが多い）
- **IMO / IMHO**
  - 「私の意見では」= 断定を避けて意見を言う前置き

## PRの状態

- **WIP**
  - Work In Progress（作業中）。まだマージしてほしくない
- **Draft PR**
  - 公式の「作業中」状態。早めに共有して相談したいときに使う
- **Ready for review**
  - レビューしてOKの状態（作業が一段落）

## 取り込み方法（マージ関連）

- **Merge**
  - ブランチの変更を取り込む（PRの取り込み）
- **Squash merge**
  - 複数コミットを1つにまとめて取り込む（履歴をスッキリさせたいとき）
- **Rebase**
  - 履歴の並びを整える（“積み直す”）。会話では「rebaseして履歴を綺麗にして」と言われがち
  - 注意: 共有済みブランチでのrebaseは運用ルール次第で避けることがある

## 変更要求の言い回し

- **Blocker**
  - これが直るまでマージできない（重要な指摘）
- **Must / Should / Nice-to-have**
  - 必須 / 推奨 / あったら良い（優先度の整理）
- **Follow-up**
  - 今回は別PRや別Issueで対応（スコープ調整）

## CI / テスト関連

- **CI**
  - 自動テスト・自動チェックの仕組み（GitHub Actions など）
- **Green / Red**
  - CIが成功（緑）/失敗（赤）
- **Flaky**
  - たまに落ちる不安定なテスト

## 責任・権限まわり

- **Approve**
  - レビュー承認（運用によって必要人数が決まる）
- **Request changes**
  - 修正要求（承認ではない）
- **CODEOWNERS**
  - ファイル/領域ごとにレビュー責任者を指定する仕組み（導入しているチームのみ）

## 「cherry-pick」ってレビューで聞く？

チームによっては「この修正だけ先にリリースへ入れたい」など、**特定の変更だけを別ブランチへ持っていく**話が出ます。
そのときは「バックポート（backport）」や「この修正をリリースブランチへ取り込む」といった言い方をすることもあります。
