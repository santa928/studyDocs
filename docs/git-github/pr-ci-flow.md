# PRの流れとCI

## この章でできるようになること

- PR（Pull Request）が「何をする場か」を説明できる
- CI が落ちたときに何が起きているか分かる
- レビュー会話の流れ（Draft→Review→Approve→Merge）が見える

## PR（Pull Request）とは

PR は「変更を取り込んでほしい」という提案であり、
**差分を見せて、合意を取って、取り込む**ための場です。

## ざっくり流れ（よくある）

1. ブランチを切って作業
2. PR を作る（必要なら Draft）
3. レビュー（コメント・質問・修正）
4. CI が通る（Green）
5. Approve
6. Merge（取り込み）

## Draft PR（作業中の共有）

「まだ完成じゃないけど、方向性を早めに相談したい」場合に Draft を使います。

## CI（自動チェック）

CI は PR や push をトリガーに、機械的にチェックします（例: テスト、Lint、ビルド）。

- **Green**: 全部OK
- **Red**: どれか失敗（テストが落ちた、ビルドできない、など）

## “CIが落ちた”と言われたら

非エンジニアが押さえるべき要点はこれだけで十分です。

- 「このPRは今マージできない状態」
- 「どのチェックが落ちたか（テスト？Lint？）」を確認する
- 必要なら「再実行できる？」を聞く（たまに一時的な失敗もある）
