# Gitとは（ざっくり一言で）

## この章でできるようになること

- Git が「何のための道具か」を説明できる
- Git が管理しているのが **ファイル**ではなく **変更の履歴**だと理解できる
- `add` と `commit` の役割の前提がつかめる

## Gitとは

Git は **変更履歴を記録していく仕組み（バージョン管理）**です。

日々の作業で「いつ、何を、なぜ変えたか」を残せるので、例えば次ができます。

- **いつでも戻れる**: 「昨日の状態に戻したい」ができる
- **差分が見える**: どこをどう変えたかを説明できる
- **同時進行できる**: 複数の作業を分けて進め、最後に合流できる

## Gitが得意なこと（会話で必要なポイント）

- **履歴（コミット）**を積み上げていく
- 履歴は **ブランチ**で分岐できる（並行作業）
- 後から **マージ**して合流できる

## Gitが管理している3つの領域（超重要）

Gitは大きく3つの「置き場所」を意識すると話がスッキリします。

- **ワークツリー**: いま編集している作業中のファイル
- **ステージ（インデックス）**: 「次のコミットに入れるもの」を選ぶ場所
- **履歴（リポジトリ）**: コミットとして確定した変更の積み重ね

この3つの関係が、`add` と `commit` の違いに直結します。

## add と commit の違い（まずは図で）

- **`git add`**: ワークツリー → ステージへ「次に入れる変更」を選ぶ
- **`git commit`**: ステージ → 履歴へ「変更を確定」して記録する

言い換えると、

- `add` は **コミットの下ごしらえ（選別）**
- `commit` は **履歴に残す（確定）**

## よくある誤解

- 「Git = GitHub」ではない
  - Git はローカルでも使える（自分のPC内で履歴管理できる）
  - GitHub は Git の履歴を **共有**したり **レビュー**するためのサービス（次章）
